# Day 3 总结

## Day 3 目标
Day 3 的核心目标是把一个“单轮问答 Demo”升级为“支持多轮上下文的 AI 陪练应用”，重点放在三件事上：

- 支持多轮会话历史
- 支持流式输出
- 提升前端产品化体验

这一天不只是继续接接口，而是开始真正进入 AI 应用开发里非常典型的一层：`Prompt + Context + Streaming + UX`。

## Day 3 做了什么

### 1. 后端从单轮消息升级为多轮上下文
原本后端只接收单条 `message`，Day 3 改成接收 `messages` 数组，让前端把会话历史一起带过来。

具体变化：

- 新增 `ChatMessageDto` 和 `ChatDto`，对消息数组做 DTO 校验
- `ChatController` 改为接收 `body.messages`
- `ChatService` 新增 `buildModelMessages()`，统一构造最终发给模型的消息列表
- 后端在真正请求模型前，自动插入系统提示词 `system prompt`
- 对历史消息做截断，只保留最近若干轮，避免上下文无限膨胀

这样做的意义是：

- 前端只负责传“用户和助手的历史消息”
- 系统角色和提示词由后端统一控制
- Prompt 策略不会散落到前端，后续更容易维护

### 2. 明确了 Prompt 管理边界
Day 3 把系统提示词抽离到了独立常量文件，由后端统一注入，而不是让前端传 `system`。

这样设计的好处：

- 更安全，前端不能随意篡改系统角色
- 更稳定，Prompt 策略集中管理
- 更贴近真实项目架构，后端掌控模型行为边界

### 3. 保留并完善了流式输出链路
前一天已经打通了基础 streaming，Day 3 在多轮会话场景里继续沿用这条链路。

后端侧：

- `ChatController` 暴露 `POST /chat/stream`
- 通过底层 `Response` 连续 `res.write()` 增量内容
- 设置 `Content-Type`、`Cache-Control`、`Connection` 等响应头
- 在异常时区分“尚未输出就失败”和“输出到一半失败”
- 在 `finally` 里确保 `res.end()`，避免前端一直挂起

模型层：

- `LlmService` 增加 `chatStream()` 方法
- 使用 OpenAI SDK 的 `stream: true`
- 通过 `for await...of` 持续消费增量 chunk

前端侧：

- 用 `fetch()` 调用流式接口
- 使用 `ReadableStream` 的 `getReader()`
- 使用 `TextDecoder` 按块解码
- 将 chunk 持续追加到最后一条 assistant 消息中

### 4. 前端从单结果展示升级为消息列表
Day 3 的前端不是只展示一个 `reply` 字符串，而是维护完整的 `messages` 数组。

核心变化：

- 新增 `ChatMessage` 类型，包含 `role`、`content`、`createdAt`
- 用户发送时，先插入一条 `user` 消息
- 同时插入一个空的 assistant 占位消息
- 流式返回过程中，持续更新最后一条 assistant 的内容
- 页面改为渲染完整消息列表，而不是单块结果区

这样做以后，应用才真正具备“对话”的形态，而不是“提交表单 -> 返回答案”的工具页。

### 5. 支持会话历史继续追问
发送新消息时，前端会把当前已有 `messages` 和新输入一起发给后端，实现多轮上下文。

这件事在 AI 应用里非常关键，因为模型本身默认不记忆历史，所谓“多轮对话”本质上是：

- 前端保存历史
- 每次请求把历史重新带上
- 后端重新组装上下文
- 模型基于完整上下文生成下一条回答

### 6. 修复了第二轮消息报错问题
在 Day 3 的多轮会话里，出现了一个典型 bug：第一条消息正常，第二条消息报错。

根因是后端 DTO 一开始只允许：

- `user`
- `system`

但第二次请求时，前端历史里已经包含上一条 `assistant` 消息，导致 `ValidationPipe` 拦截并返回 400。

后续修复为：

- DTO 改为允许 `user` 和 `assistant`
- 不允许前端直接传 `system`
- 由后端统一追加系统提示词

这个修复本身也说明了一个重要设计原则：客户端只传对话历史，系统行为由服务端控制。

### 7. 前端错误提示更可读
原来的前端只要收到非 2xx 响应，就统一提示“流式请求失败，请检查后端接口”。

Day 3 后续优化里做了改进：

- 遇到非 `ok` 响应时，优先解析后端返回的 `message`
- 如果后端返回的是数组，也会拼接展示
- 如果不是 JSON，再回退到默认错误提示

这让调试效率明显更高，尤其适合开发期排查 DTO 校验、接口异常、模型报错。

### 8. 做了一批前端产品化体验
Day 3 后期不是只追求“能跑”，而是开始往“能用、顺手、像产品”推进。

主要包括：

- 输入框自动聚焦
- `Enter` 发送，`Shift + Enter` 换行
- 会话区自动滚动到底部
- AI 输出中的 typing indicator
- 显示消息时间戳
- 支持中断生成
- 支持清空会话
- 支持重新生成上一条 AI 回复
- AI 消息支持复制
- 清空会话从原生 `confirm` 升级为自定义弹层
- 复制按钮增加默认态/成功态样式区分
- 复制成功按钮增加图标切换与动效反馈

这些内容虽然不属于“模型能力”，但在真实 AI 产品里非常重要，因为用户感知往往来自交互细节而不是接口本身。

## Day 3 涉及的技术点

### 后端技术点

- NestJS Controller / Service 分层
- DTO 设计与 `ValidationPipe`
- `class-validator` 与 `class-transformer`
- `express.Response` 直写流式响应
- HTTP streaming 基础处理
- OpenAI SDK 流式调用
- `for await...of` 异步迭代
- Prompt 管理与系统提示词注入
- 多轮会话上下文拼装
- 上下文截断策略
- 错误处理与连接关闭

### 前端技术点

- React `useState`
- React `useEffect`
- React `useRef`
- 多消息列表状态管理
- `fetch()` 调用流式接口
- `ReadableStream` / `getReader()`
- `TextDecoder` 解码流式文本
- 占位消息 + 增量拼接更新
- 自动滚动与输入框聚焦
- `AbortController` 中断请求
- 条件渲染与加载状态
- 自定义弹层交互
- 复制到剪贴板 `navigator.clipboard`
- CSS 状态样式与成功动效

### AI 应用层技术点

- 多轮对话不是“模型记住了”，而是“每次把历史重新喂给模型”
- `system / user / assistant` 三种角色分工
- Prompt 不放前端，放后端统一管理
- 上下文长度需要控制，不能无限堆积
- 流式输出能显著提升 AI 产品的主观响应速度

## Day 3 最关键的收获

如果只用一句话总结 Day 3，可以概括为：

> 把一个“能请求模型”的 Demo，升级成了一个“有上下文、有流式体验、有基础产品交互”的 AI 对话应用。

这一步的本质，是从“接 API”走向“构建 AI 应用”。

## 面试回答版

下面这部分可以直接当作你面试时的回答模板。

### 版本一：1 分钟回答
Day 3 我主要做了三件事。第一，我把原来单轮的问答接口升级成了多轮会话接口，前端会把历史消息数组发给后端，后端统一拼接 `system prompt` 和最近几轮上下文后再请求模型。第二，我把流式输出接到了多轮场景里，后端通过流式写回增量文本，前端用 `ReadableStream` 和 `TextDecoder` 持续消费并实时更新最后一条 assistant 消息。第三，我补了一些产品化能力，比如自动聚焦、Enter 发送、清空会话、自定义弹层、复制消息、重新生成和打字中状态，让它更像一个真实可用的 AI 陪练产品。

### 版本二：面试官追问“这里面最难的点是什么”
最难的点不是把接口调通，而是把多轮上下文、流式输出和前端状态更新这三件事同时处理好。因为在流式场景下，assistant 消息一开始是空的占位，后面要不断把 chunk 追加进去；而到了第二轮以后，请求体里又必须带上历史消息，否则模型就没有上下文。另外我还遇到过一个典型问题，就是后端 DTO 只允许 `user` 和 `system`，导致第二轮请求里包含 `assistant` 历史时直接被校验拦截。后来我把客户端消息角色收敛为 `user` 和 `assistant`，系统提示词统一由后端注入，这样边界就清楚了。

### 版本三：面试官追问“你在这个过程中体现了哪些工程思考”
我主要做了三层工程思考。第一层是职责边界，前端只传消息历史，后端统一管理 system prompt 和上下文裁剪，避免 Prompt 策略散落在客户端。第二层是健壮性，流式响应里区分开始输出前失败和输出中失败，并且保证连接最终能正确关闭。第三层是用户体验，我没有只停留在“返回结果”，而是继续补了自动滚动、停止生成、复制成功反馈、自定义确认弹层这类交互细节，因为 AI 产品的可用性很大程度取决于这些体验。

### 版本四：如果面试官问“Day 3 学到了什么”
我最大的收获是，AI 应用开发和普通 CRUD 不太一样，核心不只是接口和页面，而是围绕上下文管理、Prompt 组织、流式体验和用户反馈做完整闭环。模型不会天然记住历史，多轮能力是应用层自己实现的；而流式输出也不是单纯把 `stream: true` 打开就结束了，还要处理前后端的数据传递、状态更新和异常收口。Day 3 让我开始从“调用模型”转到“设计 AI 应用”这个视角。

## 可以继续扩展的方向

- 增加历史会话持久化，把内存态会话保存到数据库
- 增加 token 统计和上下文裁剪策略优化
- 给错误状态做更细粒度分类，比如参数校验失败、模型调用失败、网络失败
- 支持单条消息级别的重试，而不只是重试最后一轮
- 引入 Markdown 渲染，让 AI 回复更适合阅读
